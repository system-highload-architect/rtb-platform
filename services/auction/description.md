## 1. `services/auction/go.mod`
**Назначение:** Объявляет модуль сервиса `auction` и его зависимости.  
**Содержит:** ссылки на общие модули `pb`, `pkg`, а также клиенты Aerospike, gRPC и другие.  
Благодаря `go.work` в корне монорепо все локальные модули (`pb`, `pkg`) подтягиваются автоматически без `replace`-директив.

## 2. `services/auction/configs/dev.yaml`
**Назначение:** Конфигурация сервиса для локальной разработки.  
**Содержит:**
- `server.port` — порт gRPC (9001);
- `aerospike` — адреса, namespace, set для хранения профилей пользователей;
- `mongo` — строка подключения к MongoDB для загрузки кампаний;
- `scoring` — коэффициенты для моделей LTV, CTR/CVR, базовая ценность конверсии, параметр гео-затухания;
- `fraud.blacklist` — список IP/DeviceID, которые блокируются;
- `log` и `metrics` — настройки логирования и OpenTelemetry.

## 3. `services/auction/internal/domain/types.go`
**Назначение:** Определяет **основные сущности предметной области** (доменные типы), не зависящие от внешних библиотек и протоколов.  
**Структуры:**
- `Campaign` — рекламная кампания с полями: ID, ставка в копейках, дневной бюджет, URL креатива, координаты рекламного щита. Используется для передачи между адаптерами и движком аукциона.
- `BidEvent` — событие о результате аукциона (для отправки в Kafka / ClickHouse). Содержит идентификатор аукциона, победившую кампанию, цену, оценки скоринга (LTV, гео-фактор, ценность показа).
- `BidResponse` — результат, возвращаемый движком аукциона: ID кампании, финальная ставка, URL креатива.
- `GeoResolver` — **интерфейс**, определённый прямо в домене, чтобы не зависеть от adapters. Реализуется адаптером `geodata`.

**Зачем:** Домен ничего не знает о транспорте (gRPC), базах данных (Aerospike, MongoDB) или форматах обмена (protobuf). Его легко тестировать, подменяя интерфейсы моками.

## 4. `services/auction/internal/domain/sorter.go`
**Назначение:** Тонкая обёртка над `pkg/radixsort`, предоставляющая домену чистый метод сортировки.  
**Метод:** `SortCandidatesByEffectiveBid(bids []int64, indices []int)` — сортирует массив эффективных ставок по возрастанию, синхронно переставляя индексы. После вызова максимальная ставка находится в последнем элементе.  
**Почему отдельно:** Изолирует детали реализации (LSD Radix Sort) от логики аукциона.

---

## 5. `services/auction/internal/domain/scoring/scorer.go`
**Назначение:** Определяет **интерфейс Scorer** и структуры данных, необходимые для вычисления скора.  
**Структуры:**
- `CampaignScoringData` — минимальный набор данных кампании, нужный скореру (ID, ставка, координаты билборда, URL креатива).
- `UserFeatures` — признаки пользователя: вектор фич для модели LTV, координаты (Lat/Lng).
- `Scorer` — интерфейс с единственным методом `Score`, который принимает кампанию, пользователя и расстояние до щита, а возвращает скор и оптимальную ставку.

**Зачем:** Отделение данных скоринга от полных доменных структур предотвращает циклы импорта и позволяет легко заменять реализацию скорера.

---

## 6. `services/auction/internal/domain/scoring/predictive.go`
**Назначение:** **Реализация интерфейса `Scorer`** на основе наших аналитических пакетов (`pkg/valuation`).  
**Структура:** `predictiveScorer` содержит:
- `ltvModel` — модель прогноза пожизненной ценности пользователя (LTV);
- `impVal` — модель оценки ценности одного показа (учитывает CTR, CVR);
- `geoFactor` — гео-фактор, учитывающий расстояние до рекламного щита.

**Метод `Score`:**
1. Вычисляет LTV пользователя по вектору признаков.
2. Вычисляет базовую ценность показа.
3. Вычисляет гео-фактор (чем ближе пользователь к щиту, тем выше значение).
4. Перемножает их и получает **итоговый скор** (ценность показа для этого пользователя и этой кампании).
5. Умножает базовую ставку кампании на скор, получая **оптимальную ставку** для аукциона.

**Почему так:** Все вычисления используют готовые высокопроизводительные инструменты из `pkg`. Скорер можно заменить на другой (например, с машинным обучением), реализовав тот же интерфейс.

## 7. `services/auction/internal/domain/engine.go`
**Назначение:** **Ядро аукциона** — чистая бизнес-логика, не зависящая от инфраструктуры.  
**Функция `RunAuction`:**
- Принимает слайс кампаний, признаки пользователя, `GeoResolver` и `Scorer`.
- Для каждой кампании:
  - Получает координаты её билборда (через `GeoResolver`).
  - Вычисляет расстояние от пользователя до билборда.
  - Вызывает `scorer.Score()`, передавая данные кампании, пользователя и расстояние.
  - Сохраняет кампанию, её эффективную ставку и скор.
- Сортирует всех кандидатов по эффективной ставке (используя `sorter.go`).
- Выбирает победителя — кампанию с максимальной ставкой.
- Возвращает `BidResponse` с ID победителя, ценой и креативом.

**Зачем:** Вся логика выбора победителя инкапсулирована здесь. При изменении правил аукциона (например, второй цены) меняется только этот файл.

## 8. `services/auction/internal/ports/interfaces.go`
**Назначение:** Определяет **контракты (интерфейсы)** для внешних зависимостей, которые нужны домену и gRPC-серверу.  
**Интерфейсы:**
- `UserProfileRepo` — получение профиля пользователя по deviceID.
- `CampaignRepo` — получение активных кампаний.
- `FraudDetector` — проверка на мошенничество по IP/deviceID.
- `GeoResolver` — получение координат билборда кампании.
- `EventPublisher` — публикация событий о результатах аукциона.

Также определена структура `UserProfile` — представление профиля на уровне портов.  
**Зачем:** Благодаря интерфейсам мы можем легко заменить реальный Aerospike на заглушку для тестов, не меняя бизнес-логику.

## 9–14. Адаптеры (adapters)
Реализуют интерфейсы из `ports`, связывая домен с конкретными технологиями.

- **`aerospike/user_repo.go`** — ходит в Aerospike за профилем пользователя. Пока возвращает заглушку, но структура готова для наполнения реальной логикой (чтение bins).
- **`fraud/detector.go`** — хранит в памяти карту запрещённых IP и deviceID. Проверяет по `RWMutex`, мгновенно отвечает, не создавая нагрузки.
- **`mongodb/campaign_repo.go`** — хранит кампании в оперативной памяти (кэш). Имеет метод `Update` для периодического обновления из MongoDB. `GetActive` возвращает копию среза для безопасности.
- **`geodata/resolver.go`** — хранит координаты билбордов в памяти (ключ — ID кампании). Реализует доменный интерфейс `GeoResolver`.
- **`kafka/publisher.go`** — заглушка публикатора событий. В будущем будет отправлять `BidEvent` в Kafka.

## 15. `services/auction/internal/server/grpc.go`
**Назначение:** **gRPC-сервер**, реализующий сгенерированный из proto интерфейс `AuctionServiceServer`.  
**Структура `AuctionServer`** содержит все зависимости (репозитории, детектор фрода, резолвер, скорер, паблишер, хранилище идемпотентности, логгер).

**Метод `Bid`:**
1. Проверяет `IdempotencyKey` (через `idempotent.Store`) — повторные запросы с тем же ключом отвергаются.
2. Проверяет, не подозрительный ли пользователь (через `FraudDetector`).
3. Получает профиль пользователя из `UserProfileRepo`.
4. Получает список активных кампаний из `CampaignRepo`.
5. Формирует `UserFeatures` и вызывает `domain.RunAuction`.
6. Если есть победитель, публикует событие через `EventPublisher`.
7. Возвращает `BidResponse` в формате protobuf.

**Зачем:** Это точка входа для внешних запросов от Gateway. Он связывает все компоненты воедино, следуя потоку обработки.

## 16. `services/auction/cmd/main.go`
**Назначение:** **Точка входа** (entrypoint) сервиса. Инициализирует всё и запускает gRPC-сервер.  
**Что делает:**
- Загружает конфигурацию из YAML.
- Настраивает структурированный логгер и метрики.
- Создаёт экземпляры адаптеров (пока с заглушками).
- Собирает gRPC-сервер, передавая все зависимости.
- Запускает сервер в отдельной горутине.
- Настраивает graceful shutdown (при получении SIGTERM останавливает сервер и завершает работу).

---

## Итог: как всё работает вместе

1. Gateway получает JSON-RPC запрос от SSP → отправляет gRPC `BidRequest` в аукцион.
2. gRPC-сервер (`server/grpc.go`) проверяет идемпотентность и фрод, загружает профиль пользователя и кампании.
3. Домен `engine.go` для каждой кампании вычисляет расстояние до билборда, скорер оценивает ценность показа, и выбирается победитель с максимальной эффективной ставкой (сортировка через `radixsort`).
4. Результат возвращается в Gateway, который отвечает SSP.
5. Событие о выигравшей ставке публикуется для аналитики.
